# ADR 0094: `run` / `init` / `project create` workflow and project-profile lifecycle

## Status

Accepted

- [CLI UX Vision](../CLI-UX-VISION.md)

## Context

The repository now has a clearer CLI UX direction, but the first-run and project-profile workflow
still lacks a narrow executable decision. The main unresolved points are:

- what `operator init` owns
- how `project create` relates to committed and local profile variants
- how `run` behaves when no explicit goal is provided
- how profile precedence is applied
- whether `--agent` replaces `--allowed-agent` as the user-facing flag

These decisions materially affect onboarding, profile management, and the public CLI contract.

### Current truth

Today:

- `operator init` now creates the committed default `operator-profile.yaml`, ensures
  `operator-profiles/`, and updates `.gitignore` with `.operator/`
- `operator project create` now writes committed named profiles by default and local named profiles
  with `--local`
- profile discovery now merges committed named profiles and local named profiles
- `run --agent` is now the canonical user-facing flag
- `run` now prompts for a goal when the selected project profile does not define a default
  objective
- broader workflow areas such as ticket intake and reporting remain outside this ADR's scope

## Decision

The CLI must expose one coherent project-backed start workflow built around `init`, `run`, and
`project create`.

### `operator init`

`operator init` is the canonical first-run setup command.

It owns:

- creating `operator-profile.yaml` when absent
- creating `operator-profiles/`
- ensuring `.operator/` is gitignored
- explaining or optionally enabling `operator-history.jsonl`

It is top-level because project setup is a primary workflow, not a niche profile-management detail.

### `operator project create`

`operator project create [NAME] [--local]` is the canonical named-profile creation surface.

The storage split is:

- `operator-profile.yaml` for the committed default profile
- `operator-profiles/*.yaml` for committed named variants
- `.operator/profiles/*.yaml` for local-only named variants

`project create` writes to committed storage by default and to local-only storage when `--local` is
passed.

### Profile precedence

The effective profile precedence is:

1. CLI flags
2. local profile
3. committed named profile
4. committed default profile
5. global config defaults

List-valued CLI flags use replacement semantics rather than additive merge semantics.

### `operator run`

`operator run [GOAL]` is the canonical start command.

Its workflow rules are:

- if a goal is supplied, use it
- if no goal is supplied and the active profile has a default objective, use that
- if neither exists, prompt interactively

The user-facing agent-selection flag is `--agent`, not `--allowed-agent`.

## Consequences

- Project setup becomes one coherent user story instead of scattered command behavior.
- The repository can implement `run` without profile-resolution ambiguity.
- Named committed profiles and local overrides gain one explicit creation flow.
- The public CLI can migrate toward `--agent` without reopening profile-lifecycle debates.
- Existing `project init` style surfaces must be aliased, moved, or retired intentionally.

## Implementation notes

The current repository has implemented:

- top-level `init`
- committed vs local named profile creation
- merged named-profile lookup across committed and local storage
- `run --agent`
- interactive goal prompting when no explicit or profile-derived objective exists

## This ADR does not decide

- PM/ticket intake via `--from`
- result-reporting hooks
- provider credential management
- the exact interactive prompt wording for missing goals
- detailed file templates generated by `init`

## Alternatives Considered

### Keep setup only under `project init`

Rejected. First-run setup is primary workflow surface, not buried maintenance.

### Keep `--allowed-agent` as the long-term public flag

Rejected. It exposes an internal policy concept instead of a simple user intent.

### Treat all profiles as local-only artifacts

Rejected. The workflow vision explicitly keeps committed project defaults and committed named
variants as shared project truth.
